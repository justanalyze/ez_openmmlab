## 🤖 Supported Models

### RTMPose — Top-Down Pose Estimation
RTMPose is a **top-down** model. This means it works in two stages:

```
Stage 1 — Detect:   A detector finds each subject and draws a bounding box
Stage 2 — Estimate: RTMPose estimates keypoints inside each bounding box
```

RTMDet is the natural partner for Stage 1 — they are designed to work
together. However, there is an important distinction depending on your subject:

**Subjects that are COCO classes (person, cat, dog, horse, etc.)**
You can use a pretrained RTMDet model directly. No detection training needed.
The pretrained model already knows how to find your subject. You only need
to train RTMPose on your custom keypoints.

**Subjects that are NOT COCO classes (crayfish, custom animals, industrial parts, etc.)**
Pretrained RTMDet models have never seen your subject and cannot detect it.
You must train **both models**:
1. Train RTMDet on your subject first to get a working detector
2. Train RTMPose on your custom keypoints second

This is the complete pipeline for custom domain pose estimation:

```python
from ez_openmmlab import RTMDet, RTMPose

# Step 1 — Train your detector on your custom subject
detector = RTMDet("rtmdet_s")
detector.train(dataset_config_path="detection_dataset.toml", epochs=100)

# Step 2 — Train your pose model on your custom keypoints
pose_model = RTMPose("rtmpose_s")
pose_model.train(dataset_config_path="pose_dataset.toml", epochs=210)

# Step 3 — Run inference: pass your trained RTMDet into RTMPose
# RTMDet.train() outputs a user_config.toml in your work_dir
results = pose_model.predict(
    "image.jpg",
    det_model="runs/train/user_config.toml",           # generated by RTMDet.train()
    det_weights="runs/train/best_coco_bbox_AP_epoch_100.pth",
    det_cat_ids=[0],                                    # category ID of your custom subject
    show=True,
)
```

| Model | Speed | Accuracy | Best For |
| :--- | :--- | :--- | :--- |
| `rtmpose_tiny` | ⚡⚡⚡⚡ | ⭐⭐ | Edge devices, real-time on CPU |
| `rtmpose_s` | ⚡⚡⚡ | ⭐⭐⭐ | Balanced speed and accuracy |
| `rtmpose_m` | ⚡⚡ | ⭐⭐⭐⭐ | Production use cases |
| `rtmpose_l` | ⚡ | ⭐⭐⭐⭐⭐ | Highest accuracy, GPU recommended |

### RTMO — Bottom-Up One-Stage Pose Estimation
RTMO is a **one-stage** model. It detects subjects and estimates keypoints
in a single forward pass — no separate detector needed. This makes it
simpler to set up for custom subjects since you only train one model,
but it is generally less accurate per keypoint than the RTMDet + RTMPose
top-down pipeline.

| Model | Speed | Accuracy | Best For |
| :--- | :--- | :--- | :--- |
| `rtmo_s` | ⚡⚡⚡ | ⭐⭐⭐ | Crowded scenes, faster inference |
| `rtmo_m` | ⚡⚡ | ⭐⭐⭐⭐ | Balanced for multi-person |
| `rtmo_l` | ⚡ | ⭐⭐⭐⭐⭐ | Best multi-person accuracy |

> **RTMPose vs RTMO — which should I use?**
>
> | Scenario | Recommendation |
> | :--- | :--- |
> | Subject is a COCO class (person, animal, etc.) | RTMPose — use pretrained detector, only train keypoints |
> | Subject is NOT a COCO class, accuracy is priority | RTMPose — train both RTMDet and RTMPose |
> | Subject is NOT a COCO class, simplicity is priority | RTMO — train one model only |
> | Multiple subjects per image | RTMO — handles crowded scenes natively |
> | Highest keypoint accuracy needed | RTMPose — top-down is more precise per instance |

## 📋 Understanding COCO Pose Annotations

Before configuring your dataset, your annotation JSON must follow the
COCO keypoint format. Here is the minimum structure `ez_openmmlab` expects:

```json
{
  "categories": [
    {
      "id": 1,
      "name": "your_class",
      "keypoints": ["keypoint_0", "keypoint_1", "keypoint_2"],
      "skeleton": [[0, 1], [1, 2]]
    }
  ],
  "annotations": [
    {
      "id": 1,
      "image_id": 1,
      "category_id": 1,
      "keypoints": [x0, y0, v0, x1, y1, v1, x2, y2, v2],
      "num_keypoints": 3,
      "bbox": [x, y, width, height],
      "area": 1234.5,
      "iscrowd": 0
    }
  ],
  "images": [
    {
      "id": 1,
      "file_name": "image_001.jpg",
      "width": 640,
      "height": 480
    }
  ]
}
```

### Keypoint Visibility Flags

Each keypoint has three values: `x, y, v` where `v` is the visibility flag:

| Value | Meaning |
| :--- | :--- |
| `0` | Not labeled — keypoint does not exist in this annotation |
| `1` | Labeled but not visible — occluded or outside frame |
| `2` | Labeled and visible |

### Example — 3 Keypoint Animal

```json
"keypoints": ["head", "body", "tail"],
"skeleton": [[0, 1], [1, 2]],

"keypoints": [120, 85, 2, 145, 130, 2, 200, 180, 1]
```
This means: head at (120, 85) visible, body at (145, 130) visible,
tail at (200, 180) occluded.

## ⚙️ Dataset Configuration (pose_dataset.toml)

Your `pose_dataset.toml` must mirror your COCO annotation file exactly.
The order and names of keypoints and skeleton connections must match.

### Minimal Example — 3 Keypoints

```toml

data_root = "datasets/my_pose_project"
classes = ["your_class"]

[train]
ann_file = "annotations/train.json"
img_dir = "images/train"

[val]
ann_file = "annotations/val.json"
img_dir = "images/val"

[metainfo]

# Smaller value = stricter scoring for that keypoint
# Typical ranges: 0.025 (precise points like eyes) to 0.1 (loose points like hips)
sigmas = [0.05, 0.05, 0.05]

# One weight per keypoint — higher weight = more training emphasis
joint_weights = [1.0, 1.0, 1.0]

# --- Keypoint definitions ---
# id must start at 0 and match the index in your COCO "keypoints" list

[metainfo.keypoint_info.0]
name = "head"        # Must match COCO keypoints[0]
id = 0
color = [51, 153, 255]

[metainfo.keypoint_info.1]
name = "body"        # Must match COCO keypoints[1]
id = 1
color = [0, 255, 0]

[metainfo.keypoint_info.2]
name = "tail"        # Must match COCO keypoints[2]
id = 2
color = [255, 128, 0]

# --- Skeleton connections ---
# link names must match keypoint names defined above
# id must start at 0

[metainfo.skeleton_info.0]
link = ["head", "body"]    # Must match COCO skeleton[0]
id = 0
color = [0, 255, 0]

[metainfo.skeleton_info.1]
link = ["body", "tail"]    # Must match COCO skeleton[1]
id = 1
color = [255, 128, 0]
```

## 🏋️ Training

### Basic Training

```python
from ez_openmmlab import RTMPose

model = RTMPose("rtmpose_s")

model.train(
    dataset_config_path="pose_dataset.toml",
    epochs=210,
    batch_size=32,
)
```

> [!TIP]
> Training interrupted? Resume instantly:
> ```python
> model = RTMPose(model="user_config.toml")
> model.resume()
> ```

> [!IMPORTANT]
> **Windows users:** Always wrap in `if __name__ == "__main__":` to avoid
> multiprocessing errors with PyTorch's DataLoader.

### 🔑 RTMPose-Specific Parameters

These three parameters have the biggest impact on training quality for
custom subjects. Getting them right for your specific class is important.

#### `input_size: Tuple[int, int]` — Default: `(192, 256)`

The resolution `(width, height)` that each cropped subject is resized to
before being fed into the model. Think of it as the window the model looks
through when estimating keypoints.

**The rule: match your subject's natural aspect ratio.**

A human body is taller than it is wide — `(192, 256)` fits well.
A crayfish is wider than it is tall — `(256, 192)` fits better.
A roughly square subject — `(192, 192)` is the right choice.

Using the wrong aspect ratio forces the model to learn on distorted
subjects, which hurts keypoint accuracy.

```python
# Human body — taller than wide
model.train(dataset_config_path="pose_dataset.toml", input_size=(192, 256))

# Crayfish / fish — wider than tall
model.train(dataset_config_path="pose_dataset.toml", input_size=(256, 192))

# roughly square subject (e.g. face, insect top-down view)
model.train(dataset_config_path="pose_dataset.toml", input_size=(192, 192))
```

> [!TIP]
> Use multiples of 32 for `input_size`. The backbone divides the input
> by 32 internally, so sizes like 192, 224, 256, 288, 320 work cleanly.
> Sizes that aren't multiples of 32 may cause shape mismatches.

---

#### `feature_map_size: Optional[Tuple[int, int]]` — Default: `None` (auto)

The spatial resolution of the feature map produced by the backbone —
essentially how fine-grained the internal representation is.

When `None`, ez_openmmlab derives this automatically as
`(input_width // 32, input_height // 32)`. For most use cases
**you do not need to set this manually.**

Only override it if:
- Your subjects are very small in the image and you need finer resolution
- You are hitting memory limits and need to reduce map size

```python
# Auto (recommended for most cases)
model.train(dataset_config_path="pose_dataset.toml", input_size=(256, 192))
# → feature_map_size is automatically (8, 6)

# Manual override — finer resolution for small or detailed subjects
model.train(
    dataset_config_path="pose_dataset.toml",
    input_size=(256, 192),
    feature_map_size=(16, 12),  # 2x finer than default
)
```

---

#### `simcc_sigma: Optional[Tuple[float, float]]` — Default: `None` (auto)

RTMPose uses SimCC (Simple Coordinate Classification) to predict keypoint
locations. Sigma controls how wide or narrow the target distribution is
around each keypoint coordinate — think of it as the "blur radius" of the
training target.

When `None`, ez_openmmlab scales sigma automatically relative to
`input_size`. For most use cases **you do not need to set this manually.**

Only override it if:
- Your keypoints are very precise and small (tight sigma like `(4.0, 4.0)`)
- Your keypoints are loose or anatomically ambiguous (wider sigma like `(8.0, 8.0)`)

```python
# Auto (recommended — scales with input_size)
model.train(dataset_config_path="pose_dataset.toml", input_size=(192, 256))

# Tight sigma — for precise, small keypoints (e.g. joint centers, eye corners)
model.train(
    dataset_config_path="pose_dataset.toml",
    input_size=(192, 256),
    simcc_sigma=(4.0, 4.0),
)

# Loose sigma — for anatomically vague keypoints (e.g. body center, torso)
model.train(
    dataset_config_path="pose_dataset.toml",
    input_size=(192, 256),
    simcc_sigma=(8.0, 8.0),
)
```

## 🔍 Inference

### Subject is a COCO class (person, cat, dog, etc.)

For COCO subjects, the pretrained RTMDet detector works out of the box.
No extra configuration needed — just call `.predict()` directly.

```python
from ez_openmmlab import RTMPose

# Your trained pose model
model = RTMPose(
    model="user_config.toml",
    checkpoint_path="best_model.pth"
)

# Or use a fully pretrained model
model = RTMPose("rtmpose_m")

# RTMDet detector runs automatically in the background
results = model.predict("image.jpg", show=True)

# Access structured results
for person in results[0].keypoints:
    print(f"Keypoint coordinates: {person.xy}")      # shape: [num_keypoints, 2]
    print(f"Keypoint scores:      {person.conf}")    # shape: [num_keypoints]
    print(f"Visible keypoints:    {person.visible}") # shape: [num_keypoints]
```

---

### Subject is NOT a COCO class (custom animals, objects, etc.)

For custom subjects, the default pretrained RTMDet cannot detect your
subject. You must point `.predict()` to your own trained RTMDet model
using `det_model` and `det_weights`.

```python
from ez_openmmlab import RTMPose

pose_model = RTMPose(
    model="user_config.toml",
    checkpoint_path="best_pose.pth"
)

results = pose_model.predict(
    "image.jpg",
    show=True,
    det_model="path/to/det_user_config.toml",  # your trained RTMDet config
    det_weights="path/to/det_checkpoint.pth",  # your trained RTMDet weights
    det_cat_ids=[0],                            # category IDs to detect
)

for subject in results[0].keypoints:
    print(f"Keypoints: {subject.xy}")
```

> [!TIP]
> `det_cat_ids` controls which detected category IDs are passed to the
> pose model for keypoint estimation. Default is `[0]`.
>
> **Important:** COCO annotation category IDs start at `1`, but
> `det_cat_ids` uses **0-indexed** class positions. So if your custom
> dataset has one class, use `det_cat_ids=[0]`. If you have multiple
> classes and only want pose on the second class, use `det_cat_ids=[1]`.
>
> When in doubt, use `det_cat_ids=[0]` for single-class custom datasets.

## ⚠️ Common Mistakes

### Keypoint order mismatch
Your `pose_dataset.toml` keypoint order must exactly match your COCO JSON.

```json
// COCO JSON
"keypoints": ["head", "body", "tail"]
```

```toml
# CORRECT — same order
[metainfo.keypoint_info.0]
name = "head"

[metainfo.keypoint_info.1]
name = "body"

[metainfo.keypoint_info.2]
name = "tail"
```

```toml
# WRONG — different order, will cause silent errors
[metainfo.keypoint_info.0]
name = "tail"
```

---

### Wrong number of sigmas or joint_weights
You must have exactly one value per keypoint. Mismatches cause an error at
training start.

```toml
# 3 keypoints defined → must have exactly 3 values
sigmas = [0.05, 0.05, 0.05]        # ✅ correct
joint_weights = [1.0, 1.0, 1.0]    # ✅ correct

sigmas = [0.05, 0.05]              # ❌ wrong — only 2 values for 3 keypoints
```

---

### Skeleton link names don't match keypoint names

```toml
[metainfo.keypoint_info.0]
name = "head"   # defined as "head"

[metainfo.skeleton_info.0]
link = ["Head", "body"]   # ❌ wrong — capital H won't match "head"
link = ["head", "body"]   # ✅ correct — exact match, case sensitive
```

---

### Classes list doesn't match COCO categories

```json
// COCO JSON
"categories": [{"id": 1, "name": "crayfish"}]
```

```toml
classes = ["crayfish"]   # ✅ correct
classes = ["Crayfish"]   # ❌ wrong — case sensitive
classes = ["shrimp"]     # ❌ wrong — name doesn't match
```

## 🎯 Choosing the Right Model

| Scenario | Recommended Model |
| :--- | :--- |
| One subject per image, fast inference needed | `rtmpose_tiny` or `rtmpose_s` |
| One subject per image, accuracy matters | `rtmpose_m` or `rtmpose_l` |
| Multiple subjects per image | `rtmo_s`, `rtmo_m`, or `rtmo_l` |
| Edge device / CPU only deployment | `rtmpose_tiny` or `rtmo_s` |
| Research or highest possible accuracy | `rtmpose_l` or `rtmo_l` |
| Custom animal keypoints (few keypoints) | `rtmpose_s` — trains fast |
